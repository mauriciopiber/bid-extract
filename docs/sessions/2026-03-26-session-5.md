# Session 5 — 2026-03-26

## The Core Insight: Page-First, Structure-Second

The extraction is NOT structure-first. It's:

```
Layout → Page → What does this page show? → Extract what you see → Store per-page
                                                                        ↓
                                                              Happy path: combine pages
                                                              into final structured result
```

### Why page-first?

1. We don't know in advance what the document contains
2. Page 1 might be a ranking, page 2 might be a tabulation, page 30 might have totals
3. Each page has its own layout — its own visual structure
4. The LLM sees ONE PAGE at a time — that's what it can process accurately
5. The final combined result is a DERIVED artifact from the per-page extractions

### The mental model

Each page is like a puzzle piece. We don't know the full picture until we've extracted all pieces. But each piece is self-contained — it has a type, a layout, and data.

```
Page 1: bid_ranking    → {bidders: [{name, total}, ...]}
Page 2: bid_tabulation → {sections: [{items with bids}], bidGroupName: "Base Bid"}
Page 3: bid_tabulation → {continuation of page 2, more items}
Page 4: bid_tabulation → {bidGroupName: "Supplemental", items}
```

The combine step: take all pieces, merge by bidder name, build hierarchy.

### The UI

The UI should work the same way:
- Show each page as a card
- Each card has the PDF image left, extracted data right
- The data component adapts to the page type (ranking shows a table, tabulation shows items grid)
- At the bottom (or top): the combined result — the final merged view
- The combined result is clearly labeled as "derived from pages"

### What we got wrong before

1. Tried to force a hierarchical schema first, then display pages — backwards
2. The schema should represent WHAT WE SEE on each page, not what we think the document structure should be
3. Built UI components that don't match the data
4. Kept breaking things because the UI didn't have tests

### What we need

1. **Page extraction result** is the source of truth. It's what the LLM returned for that specific page.
2. **Combined result** is derived. If a page extraction changes, the combined result changes.
3. **UI shows both**: per-page review (for verification) AND combined view (for the final output).
4. **Tests verify the UI shows what the data contains.** Not what we hope it contains.

## Regression Problem

We spent 2+ hours going in circles:
- Fix totals → break items
- Fix items → break UI
- Fix UI → lose totals
- No tests to catch it

### Solution: test first, then change

From now on:
1. Write the test for what MUST show in the UI
2. Verify it fails
3. Fix the code
4. Verify it passes
5. Then commit

### Rule added to CLAUDE.md

"THE UI IS THE SOURCE OF TRUTH. If data exists in the DB but doesn't show in the UI, it doesn't exist."

## Timeline

We have weeks. No rush. Quality over speed. Each session should:
1. Pick ONE thing to improve
2. Write tests for it
3. Build it
4. Verify tests pass
5. Commit and stop

No more "let me also quickly change this other thing" — that's where regression comes from.
