## Eval Matrix Framework (adapted from dot-sync LOESR)

### Dimensions

- **L (Layout)**: Document page type — bid_ranking, bid_tabulation (simple, matrix, multi-section)
- **E (Extractor)**: Method — generateObject+sonnet, generateObject+opus, raw JSON, tiled extraction
- **S (Sample)**: Specific pages from specific PDFs, manually verified ground truth
- **P (Prompt)**: Versioned prompts, each with clear reason for changes
- **R (Run)**: Repeated runs for consistency measurement

### Metrics

- **Item Accuracy**: % of items correctly extracted (count match)
- **Field Accuracy**: % of fields correct (itemNo, description, unit, qty, unitPrice, extendedPrice)
- **Math Accuracy**: unitPrice × quantity = extendedPrice (within tolerance)
- **Bidder Accuracy**: correct bidder names, correct number of bidders
- **Total Accuracy**: extracted totals match stated totals
- **Consistency**: % of identical results across repeated runs

### Prompt Strategy

1. **PR1**: Solid logical base — clear instructions from glossary, no guessing
2. **PR2+**: Each version fixes specific failures found in eval
3. Every prompt version has: what it changed, why, which samples it improved

### The Loop

```
Define samples → Create ground truth → Run PR1 → Score → Find failures
  → Create PR2 (fix specific failure) → Run PR2 → Score → Compare
  → Keep winner → Find next failure → Repeat
```

No random experimentation. Each iteration has a target.

## Key Learnings

- LLM gets lazy with complex nested schemas — flat is better
- generateObject eliminates JSON parse errors entirely
- Count-first anchoring helps but generateObject matters more
- max_tokens 16384 prevents physical truncation
- Prompt needs logical structure first, dialectic refinement second
- Can't have a developer handling prompts — the system must be mechanical
